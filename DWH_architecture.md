# Glamira Data Warehouse Architecture

## Overview

This document describes the data warehouse architecture for the Glamira E-Commerce Analytics Platform.

The project processes raw MongoDB event tracking data into a structured analytical warehouse using Google Cloud Storage, BigQuery, and dbt. The final model follows dimensional modeling principles with shared dimensions and multiple fact tables for sales and user behavior analytics.

---

## End-to-End Data Flow

```text
MongoDB
    │
    ▼
Google Cloud Storage
    │
    ▼
BigQuery Raw Layer
    │
    ▼
dbt Staging Layer
    │
    ▼
dbt Intermediate Layer
    │
    ▼
dbt Mart Layer
```

---

## Layer Responsibilities

| Layer        | Responsibility                                                      |
| ------------ | ------------------------------------------------------------------- |
| Raw          | Store source data loaded from MongoDB                               |
| Staging      | Clean, rename, cast, standardize, deduplicate                       |
| Intermediate | Prepare business entities                                           |
| Mart         | Build dimensions and fact tables                                    |
| Governance   | Protect sensitive fields using Policy Tags and Dynamic Data Masking |

---

# Raw Layer

## Purpose

The raw layer stores data loaded from MongoDB into BigQuery with minimal transformation.

## Table

### raw.summary_raw

Source collection:

```text
MongoDB summary collection
```

Purpose:

* Preserve source-level event data
* Store all extracted fields as raw BigQuery columns
* Serve as the source for dbt transformations

---

# Staging Layer

## Purpose

The staging layer standardizes raw data and prepares it for business modeling.

## Model

### stg_summary

Grain:

```text
1 row = 1 source event
```

Responsibilities:

* Rename raw columns into analytics-friendly names
* Cast data types
* Convert empty strings into NULL
* Convert Unix timestamps into TIMESTAMP and DATE
* Deduplicate events using event_id
* Standardize text fields

Examples:

| Raw Column         | Staging Column     |
| ------------------ | ------------------ |
| _id                | event_id           |
| time_stamp         | event_timestamp    |
| user_id_db         | customer_id        |
| price              | product_price      |
| option_category id | option_category_id |

# Fact Tables

### Business Meaning

Represents successful checkout transactions.

### Grain

```text
1 row = 1 purchased product item within a checkout_success order
```

### Source Logic

This fact table is created by filtering successful checkout events and exploding the nested `cart_products` JSON array.

```text
checkout_success event
    │
    ▼
cart_products JSON array
    │
    ▼
one row per purchased product item
```

* device_key
* traffic_source_key

### Measures

# Fact Constellation Design

This project uses two fact tables because the source data contains two different analytical grains.

| Fact Table              | Grain                            | Purpose                         |
| ----------------------- | -------------------------------- | ------------------------------- |
| fact_sales_order_detail | 1 row per purchased product item | Sales and revenue analytics     |
| fact_user_event_detail  | 1 row per user event             | Behavioral and funnel analytics |

Both fact tables share conformed dimensions:

* dim_date
* dim_customer
* dim_product
* dim_device
* dim_traffic_source

This design avoids mixing event-level and order-item-level data in the same fact table.

# Data Governance

## Sensitive Fields

| Table        | Column        | Policy Tag |
| ------------ | ------------- | ---------- |
| dim_customer | email_address | PII_EMAIL  |
| dim_device   | ip            | PII_IP     |

## Implemented Controls

* Policy Tags
* Fine-Grained Access Control
* Dynamic Data Masking
* Role-Based Access Control

## Access Behavior

| User Type           | Result                           |
| ------------------- | -------------------------------- |
| Fine-Grained Reader | Can view raw sensitive values    |
| Masked Reader       | Can view masked sensitive values |
| Unauthorized User   | Access denied                    |

# Summary

The Glamira Data Warehouse is designed to support both sales analytics and customer behavior analytics.

The architecture follows a layered ELT approach:

```text
Raw
→ Staging
→ Intermediate
→ Mart
```

The mart layer uses a fact constellation schema with shared conformed dimensions, allowing both revenue and behavioral reporting while keeping each fact table at the correct grain.
